   1: # AGENTS.md
   2: 
   3: Agent operating guide following the DUCKMIND AI philosophy, prioritizing:
   4: 
   5: 1. **Seed → Validator → Feedback loop**
   6: 2. **Validate through end-to-end behavior, not code reading**
   7: 3. **Filesystem-first memory: all important state must be written to files**
   8: 4. **Replayable runs: clear logs, artifacts, and results**
   9: 
  10: ---
  11: 
  12: ## 1) Working Contract (5 mandatory questions)
  13: 
  14: Before executing any task, the agent must fully describe:
  15: 
  16: 1. **Input**: What does the user provide? In what format?
  17: 2. **Output**: What does the result consist of?
  18: 3. **Failure modes**: In what ways can this break?
  19: 4. **Side effects**: Which files are created/modified, which commands are run, what state changes?
  20: 5. **Permissions**: What read/write/execute access is required and where?
  21: 
  22: If any item is missing, the agent **must not guess** and must request clarification.
  23: 
  24: ---
  25: 
  26: ## 2) Standard Directory Structure for Software Agents
  27: 
  28: ```text
  29: .
  30: ├── AGENTS.md
  31: ├── README.md
  32: ├── docs/
  33: │   ├── philosophy/                  # Principles summary, architecture, ADRs
  34: │   ├── specs/                       # Seed: PRD/RFC/feature contract
  35: │   ├── scenarios/
  36: │   │   ├── holdout/                 # Independent scenarios, anti-reward-hacking
  37: │   │   └── regression/              # Regression scenarios for past failures
  38: │   └── runbooks/                    # Operations procedures and incident handling
  39: ├── agent/
  40: │   ├── prompts/                     # Prompt templates by task type
  41: │   ├── policies/                    # Guardrails, permissions, and limits
  42: │   ├── tool-contracts/              # I/O contracts for scripts/tools
  43: │   └── checklists/                  # Pre/post checklists for each run
  44: ├── src/                             # Main source code
  45: ├── tests/
  46: │   ├── unit/
  47: │   ├── integration/
  48: │   └── e2e/
  49: ├── dtu/                             # Digital Twin Universe (if external integrations exist)
  50: │   ├── twins/                       # Behavioral clones of third-party APIs
  51: │   ├── fixtures/
  52: │   └── replay/
  53: ├── scripts/
  54: │   ├── validate.sh                  # Run the full system validator
  55: │   ├── scenario.sh                  # Run a single scenario by ID
  56: │   └── replay.sh                    # Replay a previously saved run
  57: ├── runs/
  58: │   └── YYYY-MM-DD/
  59: │       └── <run-id>/
  60: │           ├── input.md             # Original human instruction
  61: │           ├── contract.md          # 5 questions + assumptions
  62: │           ├── plan.md              # Execution plan
  63: │           ├── commands.log         # Commands executed
  64: │           ├── artifacts/           # Reports, snapshots, intermediate outputs
  65: │           └── result.json          # Final state + summary
  66: └── memory/
  67:     ├── index.md                     # Current knowledge index
  68:     ├── decisions/                   # Technical decisions and trade-offs
  69:     └── summaries/                   # Multi-layer summaries (pyramid summaries)
  70: ```
  71: 
  72: ---
  73: 
  74: ## 3) Standard Operating Procedure
  75: 
  76: 1. Receive instructions from `README.md` or `docs/specs/*`.
  77: 2. Write `contract.md` (5 questions + assumptions + risks).
  78: 3. Select or create scenarios in `docs/scenarios/holdout/*` before writing code.
  79: 4. Draft a short plan in `plan.md`.
  80: 5. Execute changes in `src/` and update tests.
  81: 6. Run the validator and related scenarios.
  82: 7. Save all logs and artifacts to `runs/<date>/<run-id>/`.
  83: 8. Only close the task when holdout scenarios pass and the run is replayable.
  84: 
  85: ---
  86: 
  87: ## 4) Definition of Done
  88: 
  89: A task is only complete when all of the following are satisfied:
  90: 
  91: - A clear, unambiguous seed/spec exists.
  92: - A behavior-level validator (scenario/e2e) is included.
  93: - Results pass consistently, with no flakiness.
  94: - A complete run trace exists for others to replay.
  95: - Important technical decisions have been recorded.
  96: - No permissions were used beyond what was necessary.
